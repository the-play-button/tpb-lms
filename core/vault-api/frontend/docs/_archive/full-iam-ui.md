# TPB Vault IAM Console - Architecture et Specifications UX/UI

## 1. Positionnement Produit

### Ce que vault-api EST

- **Authorization Server OAuth 2.0** 
- Gestion coarse-grained : qui peut acceder a quelle app avec quels scopes
- Multi-tenant (Organizations)
- RBAC standard (Roles, Permissions)
- Sync infrastructure (Cloudflare Access)

### Ce que vault-api N'EST PAS

- Un systeme de permissions fine-grained metier
- Un moteur ReBAC (relations entre entites metier)
- Le prototype DECK (qui est cote app, pas cote IAM)

### Inspirations marche

- Auth0 Dashboard
- Okta Admin Console
- AWS IAM Console
- Cloudflare Zero Trust

### Limitations du marche identifiees

| Solution | Probleme |

|----------|----------|

| AWS IAM | Policies JSON illisibles, puzzle mental pour comprendre les acces |

| Okta | 15+ onglets, navigation labyrinthique |

| Auth0 | Simple au debut, cauchemar a scale |

| Azure AD | UI enterprise lourde, terminologie opaque |

**Pain points universels** :

- Aucune solution ne repond simplement a "Montre-moi TOUT ce que peut faire cet utilisateur"
- Pas d'explication des refus d'acces
- Sync infrastructure = boite noire (drift invisible)
- Audit = logs techniques, pas comprehensibles
- Pas de preview avant changement

## 5. API Endpoints Requis

### Endpoints de base

| Endpoint | Methode | Description |
|----------|---------|-------------|
| `/iam/users/:id/effective-permissions` | GET | Permissions calculees |
| `/iam/matrix` | GET | Matrice User x Scope |
| `/iam/personas` | CRUD | Gestion personas |
| `/iam/organizations/:id/scopes` | CRUD | Scopes par org |

### Endpoints Differenciateurs

| Endpoint | Methode | Description | Feature |
|----------|---------|-------------|---------|
| `/iam/explain` | POST | Expliquer pourquoi user n'a pas acces | Explain Mode |
| `/iam/explain/fix-options` | POST | Proposer solutions pour donner acces | Explain Mode |
| `/iam/graph/user/:id` | GET | Graph des acces d'un user | Access Graph |
| `/iam/graph/scope/:scope` | GET | Graph inverse (qui a ce scope) | Access Graph |
| `/iam/drift` | GET | Etat de sync avec infra | Drift Detection |
| `/iam/drift/check` | POST | Forcer verification drift | Drift Detection |
| `/iam/drift/sync` | POST | Resynchroniser | Drift Detection |
| `/iam/audit/stories` | GET | Logs en format narratif | Audit Stories |
| `/iam/audit/story/:id` | GET | Detail d'une story avec before/after | Audit Stories |
| `/iam/audit/revert/:id` | POST | Annuler un changement | Audit Stories |
| `/iam/simulate` | POST | Simuler un changement | What-If |
| `/iam/simulate/impact` | POST | Calculer impact d'une action | What-If |
| `/iam/roles/:id/description` | GET | Description NL d'un role | NL Policies |
| `/iam/roles/:id/generate-description` | POST | Generer description depuis perms | NL Policies |
| `/iam/roles/:id/suggest-permissions` | POST | Suggerer perms depuis description | NL Policies |

## 6. Livrables du Plan

1. **Spec detaillee** de chaque vue (ce document, a completer)
2. **Schema de navigation** final avec sitemap
3. **Wireframes ASCII** des 8 vues standard + 6 features differenciantes
4. **Extensions schema DB** (migration SQL)

6. **Spec UX** des 6 features differenciantes

## 7. Differenciateurs vs Marche - Resume

| Feature | Auth0/Okta/AWS | TPB Vault |
|---------|----------------|-----------|
| Comprendre les acces | Puzzle mental, listes | Access Graph interactif |
| Refus d'acces | "Access Denied" | Explain Mode + solutions |
| Sync infra | Fire & forget | Drift Detection proactive |
| Audit | Logs JSON techniques | Stories narratives + revert |
| Changements | Yolo, pas de preview | What-If Simulator |
| Policies | JSON cryptique | Natural Language + generation |

- UI de gestion des secrets (deja existante)
